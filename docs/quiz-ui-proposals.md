# クイズ画面 UI 検討資料

作成: 2026-04-11 / 作成者: MaxPL

## 背景

2026-04-10 の CEO 実機確認で、Phase 8 実装（現状 A 案のベース）に対し次の問題点が指摘された：

1. クイズ開始ボタン押下直後の反応が無く、押せたか分からない → Phase 8 で ProgressView 対応済み
2. スタート前に Wiki 本文プレビューを表示していて答えが見えてしまう → Phase 8 でプレビュー削除済み
3. **クイズ開始後、穴埋めがどこにあるか見つけられない** → Phase 8 で AttributedString ハイライト＋上部バッジで暫定対応済み

Phase 8 は 3 の暫定対応に留まっており、「長文記事だと穴埋めが埋もれて発見しづらい」という根本課題は残っている。本資料はこれを解決するための UI 方向性の候補案を比較し、CEO の判断材料にすることを目的とする。

## 解決したい本質課題

- 現在の問題位置が、本文中のどこかを**一目で**特定できること
- 長文記事（Wikipedia は 5,000 〜 20,000 文字）でもスクロール迷子にならないこと
- 文脈（穴埋め前後の文）を見ながら解答できること（Wikipedia 穴埋めクイズの本質は「文脈から推測する」こと）
- **幼児〜小学校低学年でも迷わず進められる**こと（初期ターゲットユーザー）

## 4 案 比較早見表

| 観点 | A案: 本文全体表示型 | B案: 1問1画面集中型 | C案: カード分割型 | D案: チャット型 |
|---|---|---|---|---|
| 現在位置の見つけやすさ | △（ハイライトで強調中） | ◎（現問題のみ大表示） | ◎（1穴埋め=1カード） | ◎（最新問題が最下段） |
| 文脈把握 | ◎（全文見える） | ○（周辺文のみ） | △（抜粋のみ） | △（過去問題が上へ流れる） |
| 迷子回避 | △ | ◎ | ◎ | ◎ |
| 没入感／ゲーム感 | △ | ○ | ○ | ◎ |
| 幼児への優しさ | △（情報量多い） | ◎（ワンタスク） | ◎（スワイプUX） | ◎（LINE慣れ） |
| 長文記事との相性 | ◎ | ○ | ◎ | △ |
| 答え合わせの振り返り | ◎（本文上に結果残る） | △（問題間遷移が必要） | ○（カード戻れる） | ◎（会話ログ） |
| 実装コスト | **低**（Phase 8 既存） | 中 | 中〜高（抜粋ロジック） | 中 |
| 既存ロジック流用 | 100% | 90% | 70%（抜粋抽出追加） | 85% |

凡例: ◎ 非常に良い / ○ 良い / △ 課題あり

## A 案: 本文全体表示型（Phase 8 現状の強化版）

### コンセプト
記事本文全体を常時表示しつつ、現在の穴埋めを強ハイライトする。現行の延長線。

### 画面イメージ
```
┌───────────────────────────────┐
│ ⏱ 00:42   問 3/10      [ギブアップ] │
├───────────────────────────────┤
│ ┌───────────────────────────┐ │
│ │    いまの もんだい          │ │
│ │    問 3  [カタカナ語]       │ │
│ └───────────────────────────┘ │
├───────────────────────────────┤
│ 東京タワー(とうきょうタワー)は、  │
│ 東京都港区芝公園にある電波塔。    │
│ 正式名称は 日本電波塔(にほん...  │
│ 高さは【333】メートルで、        │
│ 完成当時は 自立式鉄塔として       │
│ 世界一の高さだった。 ━━━━━━━   │
│ 設計は ━━━━━━━━ が担当し...  │
│ 塔の愛称として『███ 3 ███』が │ ← 現在ここ
│ 広く親しまれており、...          │
│ （スクロールして続く）            │
├───────────────────────────────┤
│ 問3（カタカナ語）                │
│ [ ここに答えを入力      ] [決定]  │
│ [ヒント(1文字目)]  ヒント使用: 0  │
├───────────────────────────────┤
│ ✓ 問1 タワー                    │
│ ✓ 問2 333                       │
└───────────────────────────────┘
```

### メリット
- **文脈が完全に見える**：前後の文を全て参照できるので推測が容易
- **答え合わせがその場で本文に反映される**：緑字で正解、赤打消しで不正解が文中に残る → 学習効果が高い
- **実装済み**：Phase 8 で完成しており追加コストゼロ
- 大人ユーザー・長文記事との相性が最高

### デメリット
- 長文記事（例：10,000 文字）だと現在問題を探すのにスクロールが必要
- バッジで番号は示されるが「本文のどこか」は分からない
- 幼児にとっては情報量が多すぎて圧倒されがち

### 強化オプション（A 案採用時）
- **自動スクロール**：現問題へ `ScrollViewReader.scrollTo(anchor: .center)` で自動追従
- **「ジャンプ」ボタン**：バッジタップで現在問題にスクロール
- **ミニマップ**：右端に縦ゲージで全体中の位置をインジケータ表示

### 実装コスト
- 自動スクロール追加のみなら **1-2 時間**
- ミニマップまで入れて **半日**

## B 案: 1 問 1 画面集中型

### コンセプト
画面を「周辺文脈ゾーン」と「現問題ゾーン」の 2 段構成にする。ユーザーは 1 問ずつ集中して解く。過去問題・未来問題は画面から消える。

### 画面イメージ
```
┌───────────────────────────────┐
│ ⏱ 00:42   問 3/10      [ギブアップ] │
├───────────────────────────────┤
│  ～前後の文脈～                  │
│                                │
│ 「…完成当時は 自立式鉄塔として    │
│ 世界一の高さだった。             │
│                                │
│ 塔の愛称として                   │
│                                │
│ 【 ？ ？ ？ ？ 】                │ ← 大きく
│                                │
│ が広く親しまれており、            │
│ 観光地としても…」                │
├───────────────────────────────┤
│  問3（カタカナ語）                │
│ [                        ] [決定] │
│ [ヒント] ◀前 次▶                │
└───────────────────────────────┘
```

### メリット
- **現在問題が画面中央に超大きく表示される**：絶対に見失わない
- **情報量が絞られる**：幼児でも認知負荷が低い
- **集中できる**：他の問題に気を取られない
- **スマホ縦画面と相性が良い**

### デメリット
- 全体文脈が見えにくい（周辺 2-3 文のみ）→ 遠くの文から推測する問題は難しくなる
- 「前後 N 文」のロジックをどう決めるか要設計（穴埋めから何文取るか）
- 答え合わせ時に全体像を見る別画面が必要

### 実装コスト
- 周辺文抽出ロジック：穴埋め位置から前後 N 文（句点区切り）を切り出す
- 新 View 作成：1-2 日
- 既存 QuizViewModel は流用可能（currentIndex の進行管理はそのまま）

## C 案: カード分割型

### コンセプト
クイズ全体を「カードの束」に変換する。1 カード = 1 穴埋め + その周辺の 2-3 文。ユーザーはスワイプで次のカードへ進む。

### 画面イメージ
```
┌───────────────────────────────┐
│ ⏱ 00:42   問 3/10   ●●○○○○○○○○ │
├───────────────────────────────┤
│                                │
│   ┌─────────────────────┐     │
│   │                     │     │
│   │  設計は内藤多仲らが   │     │
│   │  担当し、塔の愛称と   │     │
│   │  して                │     │
│   │                      │     │
│   │   【 ？？？？ 】      │     │
│   │                      │     │
│   │  が広く親しまれて     │     │
│   │  おり...              │     │
│   │                      │     │
│   └─────────────────────┘     │
│       (カード 3/10)             │
│                                │
├───────────────────────────────┤
│ [ ここに答えを入力    ] [決定]    │
│ ◀ スワイプで前後       [ヒント]  │
└───────────────────────────────┘
```

### メリット
- **1 問 1 カードで圧倒的に見つけやすい**
- **スワイプ UX** はスマホユーザーに馴染みがある（Tinder/Instagram 世代）
- **カードを戻れる**ので復習しやすい
- **ゲーム感**：カードをめくる感覚がクイズの楽しさと合う

### デメリット
- 文脈抽出ロジックが複雑（どこからどこまでを 1 カードにするか）
- 文脈が狭まるので、広範囲参照が必要な問題は解けない
- 従来の「記事を読む」体験からは離れる
- カードをまたいで答えが関連する問題に対応しにくい

### 実装コスト
- 文脈抽出ロジック新規：半日
- `TabView(.page)` ベースのカードスワイプ UI：1 日
- 合計 **2-3 日**

## D 案: チャット型

### コンセプト
問題がチャットの吹き出しのように上から順に現れ、下の入力欄に答えを打ち込む。解答済み問題は会話ログのように残る。

### 画面イメージ
```
┌───────────────────────────────┐
│ ⏱ 00:42    問 3/10             │
├───────────────────────────────┤
│                                │
│ 🤖 問1:「とうきょう___」         │
│    カタカナ語！                   │
│                                │
│              あなた: タワー 😊  │
│              ┌──────────┐      │
│              │ ✓ 正解！   │      │
│              └──────────┘      │
│                                │
│ 🤖 問2: 高さは ___ メートル？    │
│                                │
│              あなた: 333 😊     │
│              ┌──────────┐      │
│              │ ✓ 正解！   │      │
│              └──────────┘      │
│                                │
│ 🤖 問3: 塔の愛称として【 ___ 】  │
│    カタカナ語！                   │
│                                │
├───────────────────────────────┤
│ [ ここに答えを入力    ] [送信]    │
└───────────────────────────────┘
```

### メリット
- **LINE/Discord 慣れしたユーザーには直感的**
- **達成感**：正解 → 次の問題が出る流れがテンポ良い
- **会話ログ**として過去問題が自然に残る → 振り返りやすい
- **現在問題は常に最下段**なので見失わない

### デメリット
- Wikipedia の**本文構造が失われる**：文章を読む体験ではなくなる
- 各問題に対して「問題提示用の文」をどう生成するか要設計
- 前後の文脈参照ができないので、難問（広範囲参照）に不向き
- 学習用途より娯楽用途の体験に近づく

### 実装コスト
- 問題メッセージ生成ロジック：半日
- スクロール自動追従の吹き出し UI：1 日
- 合計 **2 日程度**

## PL としての推奨

### 第 1 候補: **A 案 + 自動スクロール強化**（最小コストで最大効果）

**理由:**
1. Phase 8 の実装資産をそのまま活かせる（コスト 1-2 時間）
2. 「本文を読みながら解く」という Wikipedia 穴埋めクイズの本質を失わない
3. `ScrollViewReader.scrollTo` で現問題を自動で画面中央に持ってくれば、「どこか分からない」問題は解消する
4. さらにミニマップを加えれば、長文記事でも位置把握できる

### 第 2 候補: **B 案**（幼児・初心者に寄せる場合）

ターゲットユーザーが「幼児〜小学校低学年」寄りなら B 案が最適。情報量を絞って 1 問ずつ集中させる方が、迷子率は下がる。ただし新規実装が多く 1-2 日かかる。

### 第 3 候補: C / D 案

C 案・D 案はゲーム性を強化できるが、Wikipedia 穴埋めの学習要素が薄まる。また抽出ロジックの設計に不確実性がある。採用するなら A/B で試した後の Phase 2 として検討するのが安全。

## 質問事項（CEO 判断材料）

1. **ターゲットユーザー層**：幼児〜小学校低学年の使いやすさを最優先か、小中学生以上の学習体験を重視するか
2. **クイズの本質**：「Wikipedia 記事を読む体験」を残すか、「ゲーム的なクイズ」に寄せるか
3. **実装期間**：今日中 / 数日 / 1 週間、どの時間軸で次バージョンをリリースしたいか
4. **A 案強化で十分か、B 案以降の新規 UI を試すか**

- C/D 案採用 → 文脈抽出ロジック設計から（3 日以上）
